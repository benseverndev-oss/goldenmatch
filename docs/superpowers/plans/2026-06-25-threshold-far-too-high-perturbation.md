# `threshold_far_too_high` Gym Perturbation Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add one additive gym perturbation (`threshold_far_too_high`) that drives the primary matchkey threshold well beyond the score valley so the config-suggestion kernel's `lower_threshold` (dip) rule fires, applies, recovers F1, and survives the live health-proxy gate — turning the dip fix's accuracy value into a standing, measured regression asset in the `gym-gate` CI job.

**Architecture:** Additive change in three layers. (1) Add a `_apply_threshold_far_too_high` mutator (reusing the existing `_applies_threshold_too_high` guard) and append one `Perturbation` to `CATALOG` in `scripts/suggest_quality/perturbations.py`. (2) The dip rule's recovery only materializes under `GOLDENMATCH_SUGGEST_FULL_DIST=1` (the env gate at `adapter.py:424` that turns on full pre-threshold score-distribution computation), but the `bench-suggest-quality.yml` job does NOT currently set it — so the workflow is amended to enable FULL_DIST for the gym steps only (scorecard gate untouched) and to re-bless + commit the gym baseline back to the branch on a `mode=gym-bless` dispatch. (3) The blessed `gym_scorecard.json` is regenerated by CI under FULL_DIST and committed, locking the new `recovery_pct_live (~0.75)` in as a `gym-gate` regression floor. The gentle `threshold_too_high` (+0.10) is left untouched.

**Tech Stack:** Python (dataclasses, `copy.deepcopy`), pytest, the `scripts.suggest_quality` gym harness (`cli.py` `gym` / `gym-bless` / `gym-gate` modes), GitHub Actions (`bench-suggest-quality.yml`), the in-tree native config-suggestion kernel.

## Why FULL_DIST is load-bearing (read before Task 2/3)

- `GOLDENMATCH_SUGGEST_FULL_DIST=1` is read at `packages/python/goldenmatch/goldenmatch/core/suggest/adapter.py:424`. It enables full pre-threshold score-distribution computation, which the `lower_threshold` (dip) rule needs to locate the valley. WITHOUT it the dip rule cannot fire, so `threshold_far_too_high` recovers ~0 (not ~0.75).
- `gym-gate` (`cli.py:776`) RE-RUNS the catalog live in the same invocation and FAILS if any blessed built-rule pair's `recovery_pct_live` drops more than `RECOVERY_GATE_TOL = 0.05` vs the blessed value.
- Therefore: if the baseline blesses `0.75` for the far pair but CI's `gym-gate` re-runs WITHOUT FULL_DIST, it recomputes ~0 → drop of 0.75 → **gate goes RED**. The fix (Task 2) is to enable FULL_DIST for the gym steps so bless and gate are self-consistent.
- `gym-gate` gates FOUR things (`cli.py:838-882`), not just live: per-pair `recovery_pct_live` AND `recovery_pct_raw` (both tol 0.05), plus `headline_live` AND `headline_raw`. The existing committed baseline (blessed at FULL_DIST=0) has `headline_live = 0.0` (all rules' `mean_recovery_pct_live = 0.0`) but NON-zero raw values (e.g. `ncvr_synthetic/bad_freetext_scorer` raw = -8.345, `threshold_too_low` raw = +0.723, `headline_raw` = -2.259).
- **The transient-red risk (one push-triggered run before the bless lands):** when Tasks 1+2 are pushed, the push event runs `gym-gate` under FULL_DIST=1 (Task 2 made it unconditional) against the FULL_DIST=0 baseline — exactly once. The EXPECTED outcome is PASS, because:
  - Live: recomputed live recovery is `>= 0 = baseline` for every blessed pair → never a drop.
  - Raw: FULL_DIST only changes the dip rule's valley computation; non-dip built-rule pairs (`threshold_too_low`/`raise_threshold`, `bad_freetext_scorer`/`swap_scorer`) emit byte-identical suggestion sets either way, so their raw recovery is unchanged. The gentle `threshold_too_high` (+0.10 → 0.90) stays within `DIP_MIN_GAP` of the ~0.875 valley (gap 0.025 < 0.05) so the dip rule stays quiet → its raw is unchanged too.
  - The NEW `threshold_far_too_high` pair enters the recomputed `headline_raw`/`headline_live` mean; per the probe its live recovery is +0.75 (pulls live UP) — raw for an egregious-damage pair can be negative and could nudge `headline_raw` down.
- **This raw reasoning is EXPECTED but first VERIFIED by the CI bless run itself, not pre-proven** (verifying locally would reintroduce the heavy FULL_DIST run the CI route is meant to avoid). So the plan treats the push-triggered run as NOT the source of truth: Task 3 best-effort cancels it and the `gym-bless` dispatch (which writes a fresh baseline, then gates against it → always self-consistent) is authoritative. If the push run reds anyway, it self-heals the moment the bless commits the FULL_DIST baseline. The final state — a green `gym-gate` against the committed FULL_DIST baseline — is what the done-criteria require.

```python
def _apply_threshold_far_too_high(config: GoldenMatchConfig) -> GoldenMatchConfig:
    """Raise primary weighted matchkey threshold by 0.18, ceiling 0.99.

    The EGREGIOUS variant of threshold_too_high: +0.18 lands beyond
    DIP_MIN_GAP of the score valley (e.g. 0.80 -> 0.98 on ncvr_synthetic,
    valley ~0.875), so the dip lower_threshold rule actually fires and
    recovers matches -- unlike the gentle +0.10, which stays within the gap
    and (correctly) emits nothing.
    """
    new = copy.deepcopy(config)
    mk = _primary_weighted_mk(new)
    if mk is None:
        return new
    mk.threshold = min(0.99, mk.threshold + 0.18)
    return new
```
