# Document Ingest TS surface (WASM kernels + parity) — design

**Goal.** Expose the `documents-core` kernels (5 wasm-exported fns: schema validate/round-trip,
response parse, extract-instruction, suggest-prompt, record normalize) in the TypeScript
`goldenmatch` package via `documents-wasm`, with a
TS==Rust==Python parity test over the shared corpus. This is the payoff of building documents-core:
the deterministic document-ingest logic is ONE Rust source of truth across Python, Rust, and TS — no
reimplementation, no drift.

**Builds on:** `documents-core` + `documents-wasm` (the wasm-bindgen wrapper exporting the 5 kernel
fns) + the Python parity corpus `documents_corpus.jsonl` — all merged to main.

**Scope (settled in brainstorming): WASM kernels + parity ONLY.** Rasterization, the VLM call, and
an end-to-end TS `ingestDocuments` are NOT in scope (that is the deferred "full native pipeline"
option, and it is I/O, not parity-relevant). This surface makes the KERNELS available + proven, which
is the documents-core thesis.

## The precedent (mirror it exactly)

`autoconfig-wasm` / `suggest-wasm` -> TS is the established pattern:
- `scripts/build_<c>_wasm.mjs` builds the wasm crate via `wasm-pack build --target web`, strips the
  async init glue (neutralize `import.meta.url`, re-export only `initSync` — edge-safe, no `node:*`),
  writes `src/core/_wasm/<c>WasmBindings.{js,d.ts}` + `<c>WasmBytes.ts` (base64-inlined `.wasm`), and
  copies the golden corpus into `tests/parity/fixtures/<c>/`.
- **The generated outputs are COMMITTED**, so `tsc`/`vitest`/`tsup` need NO rust toolchain — only the
  regen step (`node scripts/build_<c>_wasm.mjs`) needs `wasm-pack` + the `wasm32-unknown-unknown`
  target.
- `src/core/<c>Wasm.ts` is a sync `initSync`-based loader exposing camelCase TS wrappers over the
  snake_case JSON kernels.
- `tests/parity/<c>-core.parity.test.ts` runs the TS kernels over the golden vectors and asserts
  byte-identical to the shared oracle.

## The kernels exposed (`documentsWasm.ts`)

The `documents-wasm` crate exports (string in / string out, snake_case JSON):
`schema_validate`, `parse_message_text`, `extract_instruction`, `suggest_prompt`, `normalize_record`.
`documentsWasm.ts` wraps them as camelCase TS fns with typed signatures:
```ts
validateSchema(schema: TargetSchema): TargetSchema     // schema_validate (round-trip canonical form)
parseMessageText(resp: unknown): string                // parse_message_text (throws on truncation/malformed)
extractInstruction(schema: TargetSchema): string       // extract_instruction
suggestPrompt(): string                                // suggest_prompt
normalizeRecord(values, confidence, schema): {values, confidence}  // normalize_record
```
JSON.stringify at the boundary (the kernels take JSON strings); JSON.parse the result. Types
(`TargetSchema`, `SchemaField`, `FieldKind`) mirror the Python/Rust contract. A JsError from a kernel
becomes a thrown JS `Error`.

## Corpus sharing (one source of truth)

The parity corpus is `packages/python/goldenmatch/tests/parity/documents_corpus.jsonl` (generated by
documents-core's `gen_documents_corpus.py` from the pure-Python impls, and already asserted ==native
in CI's Python parity lane). `build_documents_wasm.mjs` COPIES it into
`tests/parity/fixtures/documents/` (same as `build_suggest_wasm` copies its golden vectors), so the TS
parity test runs against the identical vectors Python + Rust do. Rows are `{kernel, input, expected}`.

**Row `kernel` label -> TS wrapper (the dispatch map — do NOT assume it's the Rust fn name):**
`schema` -> `validateSchema`, `parse` -> `parseMessageText`, `prompt_extract` -> `extractInstruction`,
`prompt_suggest` -> `suggestPrompt`, `normalize` -> `normalizeRecord`. (Canonical map:
`_KERNEL_SYMBOL` in the Python `test_documents_parity.py`.)

**Comparison convention (MUST mirror the Python oracle `_run_native` in `test_documents_parity.py`):**
- `schema`, `prompt_*`, `parse` success: compare parsed JSON / the string against `expected.ok`.
- **`normalize`: the corpus stores `expected.ok` as ORDERED `[col, val]` pair arrays keyed to the
  schema's field order (NOT the kernel's raw `{values, confidence}` object).** The TS test must
  reshape the wasm kernel's object output into `[[c, values[c]] for c in schema.fields]` (and likewise
  for confidence) BEFORE comparing — exactly as `_run_native` does with
  `cols = [f["name"] for f in input.schema.fields]`. A direct `JSON.parse(...) === expected` will fail
  every normalize row. This is the plan's #1 correctness trap.
- error rows (`expected.error == true`): assert the TS kernel THROWS (error-by-outcome), not an exact
  message.

## Parity assertion

For each corpus row, `documents-core.parity.test.ts` calls the TS-WASM kernel and asserts its output
equals `expected`. Green here means **TS == the committed corpus == Rust == Python** — the four
kernels are byte-identical across all three languages. (Error rows: assert the TS kernel throws, per
the corpus's error-by-outcome convention.)

## Build reality (flagged for the plan)

- Regenerating the bindings needs `wasm-pack` + the `wasm32-unknown-unknown` target (exFAT-fiddly; may
  not be installed on this box). If the regen can't run locally, the plan still authors
  `documentsWasm.ts` + the parity test + the build script + CI wiring; **CI is the authoritative
  build+parity gate** (the `documents_wasm` job regenerates and runs the test). The box OOMs on full
  TS builds; run only the single parity test file locally where possible, else rely on CI.
- The committed-generated-outputs design means once the bindings exist (from a successful regen, local
  or CI), the parity test runs with no toolchain.

## Scope (YAGNI)

**In:** the build script + generated bindings/bytes + `documentsWasm.ts` (5 wrappers) + the parity
test + the subpath export + the CI `documents_wasm` job. **Out:** rasterization, the VLM call,
end-to-end TS `ingestDocuments`/`suggestSchema` (the deferred full-native option), any api_parity
manifest change (this adds no MCP tool / CLI / A2A skill — it is a library kernel module).

## Open risks

- **wasm-pack availability / exFAT build:** the single biggest execution risk. Mitigation: commit the
  build script + TS + test authored correctly; let CI's `documents_wasm` job do the authoritative
  regen + parity run. Do NOT commit hand-written/faked bindings — either a real regen produced them or
  CI regenerates.
- **Glue-format drift:** `build_suggest_wasm.mjs` hard-asserts on the wasm-bindgen glue's exact
  strings (throws if the format changed). Mirror that guard so a wasm-bindgen version bump fails loud,
  not silently.
