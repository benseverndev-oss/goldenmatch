# goldenpipe profile_complexity cross-surface parity — design

**Status:** approved (design gate)
**Date:** 2026-07-07
**Builds on:** C2 (#1553) — the TS `profileComplexity` glue. Closes the null-density parity gap C2 deferred.

## 1. Goal & rationale

Prove TS `profileComplexity` reproduces Python `profile_complexity`'s null density on a shared, hand-authored vector — the parity confidence the "null-heavy pipe_parity fixture" was really after, without its problems.

**Why not a pipe_parity fixture** (recorded so we don't relitigate): `low_confidence` and `default` produce **identical stages** (`[scan, flow, dedupe]`), differing only in `confidence` (0.3 vs 0.7); pipe_parity observes only status/stages/counts, never confidence. So a null-heavy pipe_parity case below 100k rows is byte-identical to a `default` case — it tests nothing about null density. Null density's only observable consequence is the RED→refuse path at ≥100k rows, which doesn't fit the fixture shape and would require changing `parseCsv` (empty cell `""` → null) with blast radius into the stages. A direct `profile_complexity` vector-parity check delivers the real value with none of that.

## 2. What's being compared (and the one real constraint)

Python `profile_complexity(ctx)` (`autoconfig_glue.py:61`): `df.null_count()` per column / `n_rows` → `max_null_density`, `mean_null_density`; `df is None` or `n_rows == 0` → zeros.

TS `profileComplexity(rows)` (`autoconfigGlue.ts:40`): per-column null count (`v === null || v === undefined`) / `nRows` → same; empty → zeros.

**Constraint — uniform keys.** Python derives columns from the Polars frame schema (**union** of all row keys); TS derives them from `Object.keys(rows[0])` (**first row only**). For rows with *varying* keys they diverge (verified: `[{a,b},{a,b,c}]` → Python `['a','b','c']`, TS `['a','b']`). For **uniform-key rows** (every row has the same keys) they agree — and that is the CSV-real case (`parseCsv` gives every row the header keys). **Every vector case MUST use uniform keys.** This is documented in the vector header and is the parity contract's precondition.

Null mapping: JSON `null` → Polars null (Python) and JS `null` (TS) — both counted. (No missing keys, by the uniform-key rule.)

## 3. The vector

`packages/rust/extensions/goldenpipe-core/tests/vectors/profile_complexity.json` — the canonical cross-surface vectors dir the TS parity tests already read from. Header comment states: **Python+TS glue-parity only; NO Rust leg** — `profile_complexity` is impure host glue (Polars in Python, `Row[]` in TS), not part of the portable `goldenpipe-core` decision core, so `golden_vectors.rs` does not (and must not) replay it.

Format (mirrors the other vector files):
```json
[
  {"comment": "...", "input": {"rows": [ {...}, ... ]}, "expected": {"max_null_density": 0.0, "mean_null_density": 0.0}}
]
```

Cases (uniform keys; float values are clean binary fractions so compares are exact; expecteds generated by running real Python `profile_complexity` on the box):
- **no nulls** → `{max: 0.0, mean: 0.0}`
- **one null-heavy column** (`a`: 1/4 null, `b`: 3/4 null) → `{max: 0.75, mean: 0.5}`
- **all-null column** (`a`: 1.0, `b`: 0.0) → `{max: 1.0, mean: 0.5}`
- **single row, no null** → `{max: 0.0, mean: 0.0}`
- **single row, all null** → `{max: 1.0, mean: 1.0}`
- **empty rows** (`{"rows": []}`) → `{max: 0.0, mean: 0.0}`

**Location contingency:** if any CI gate enumerates `tests/vectors/*.json` and expects every file to be Rust-replayed (verify in the plan — `golden_vectors.rs` hard-codes family names, so an extra file is inert there), fall back to committing the vector under `packages/typescript/goldenpipe/tests/fixtures/` and adjust both tests' read paths. Default: the goldenpipe-core vectors dir.

## 4. Tests (both call the real function in-language — no bridge fns, no emitter)

Because both surfaces call `profile_complexity` / `profileComplexity` directly (no FFI/string boundary like the Rust/native legs needed), the slice is just the vector + two replay tests.

- **Python (box-runnable)** — `packages/python/goldenpipe/tests/test_profile_complexity_parity.py`: read the vector (relative path up to repo root, then into the core vectors dir), for each case build `PipeContext(df=pl.DataFrame(rows))` (empty rows → `pl.DataFrame([])`), call `profile_complexity(ctx)`, assert `max_null_density`/`mean_null_density` match `expected` (exact; `pytest.approx` guard on mean).
- **TS (CI-only)** — `packages/typescript/goldenpipe/tests/parity/profile-complexity-parity.test.ts`: read the same vector via the established `../../../../rust/extensions/goldenpipe-core/tests/vectors/profile_complexity.json` cross-read, for each case call `profileComplexity(rows as Row[])`, assert `maxNullDensity`/`meanNullDensity` with `toBeCloseTo(expected, 10)`.

Both replay the SAME committed vector → both prove they meet the contract → transitively TS == Python.

## 5. Box / CI split

- **Box-verified:** authoring the vector (run real Python `profile_complexity` to generate expecteds) + the Python replay test (`pl.DataFrame` + `profile_complexity`, no goldenflow/goldenmatch needed).
- **CI-only:** the TS replay test (vitest OOMs the box).

## 6. Non-goals

- No `parseCsv` null-semantics change; no null-heavy pipe_parity fixture; no ≥100k refuse-path test.
- No Rust/WASM leg (profiling is impure host glue, not in the portable core).
- Varying-key `Row[]` parity is explicitly out (Python union vs TS first-row) — documented as the precondition, not tested (it's not the CSV-real case).
